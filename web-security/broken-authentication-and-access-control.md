# Broken Authentication & Access Control

> Course: Web Application Security Course (Arabic)  
> Lesson: 2 - Broken Authentication & Access Control  
> Topic: Web Security / Authentication / Authorization

---

## الهدف من الدرس

فهم الفرق بين **Authentication** و **Authorization**، ومعرفة كيف تؤدي الأخطاء في تسجيل الدخول أو التحقق من الصلاحيات إلى ثغرات خطيرة داخل تطبيقات الويب.

---

## 1. Authentication

**Authentication** يعني التحقق من هوية المستخدم.

مثال:

- إدخال اسم المستخدم وكلمة المرور
- تسجيل الدخول باستخدام OTP
- تسجيل الدخول باستخدام 2FA

إذا كان التطبيق لا يتحقق من الهوية بشكل صحيح، فقد يحدث **Broken Authentication**.

---

## 2. Broken Authentication

تحدث هذه الثغرة عندما يستطيع المهاجم تجاوز أو إساءة استخدام نظام تسجيل الدخول.

أمثلة شائعة:

- كلمات مرور ضعيفة
- عدم وجود حماية ضد التخمين المتكرر
- رسائل خطأ تكشف هل المستخدم موجود أم لا
- Session لا تنتهي بعد تسجيل الخروج
- عدم استخدام 2FA في الحسابات الحساسة

---

## 3. Authorization / Access Control

**Authorization** يعني تحديد ماذا يستطيع المستخدم فعله بعد تسجيل الدخول.

مثال:

- مستخدم عادي يستطيع مشاهدة حسابه فقط
- المدير يستطيع إدارة المستخدمين
- موظف الدعم يستطيع الوصول لجزء محدد فقط

إذا استطاع مستخدم الوصول إلى بيانات أو وظائف لا يملك صلاحية لها، فهذا يسمى **Broken Access Control**.

---

## 4. Broken Access Control

تحدث هذه الثغرة عندما لا يطبق التطبيق الصلاحيات بشكل صحيح.

أمثلة شائعة:

- مستخدم عادي يدخل صفحة Admin
- تغيير ID في الرابط للوصول إلى بيانات شخص آخر
- تنفيذ وظيفة إدارية من حساب عادي
- الاعتماد على إخفاء الزر في الواجهة فقط بدون تحقق في الخادم
- السماح بتعديل الطلبات من Burp Suite بدون فحص الصلاحيات

---

## 5. الفرق المهم

| المفهوم | المعنى | مثال |
|---|---|---|
| Authentication | من أنت؟ | تسجيل الدخول |
| Authorization | ماذا يُسمح لك أن تفعل؟ | هل يمكنك دخول صفحة Admin؟ |

---

## 6. التأثير الأمني

قد تؤدي هذه الثغرات إلى:

- الوصول إلى حسابات مستخدمين آخرين
- كشف بيانات حساسة
- تعديل أو حذف بيانات بدون صلاحية
- الوصول إلى لوحة تحكم الإدارة
- رفع مستوى الصلاحيات من مستخدم عادي إلى Admin

---

## 7. طريقة الفهم والتطبيق الآمن

طبّق فقط داخل بيئات تدريب قانونية مثل:

- PortSwigger Web Security Academy
- TryHackMe
- OWASP Juice Shop
- DVWA

خطوات التعلم الآمن:

1. افهم وظيفة تسجيل الدخول.
2. جرّب الفرق بين مستخدم عادي ومستخدم بصلاحيات أعلى داخل اللاب.
3. راقب الطلبات باستخدام Burp Suite.
4. لاحظ أين يتم إرسال User ID أو Role أو Session Cookie.
5. تحقق هل الخادم يمنع الطلب غير المصرح به أم يعتمد فقط على الواجهة.
6. اكتب ملاحظاتك بصيغة تقرير مختصر.

---

## 8. الحماية

لتقليل هذه الثغرات يجب:

- تطبيق فحص الصلاحيات من جهة الخادم وليس الواجهة فقط
- استخدام مبدأ أقل صلاحية Least Privilege
- منع التخمين المتكرر بكلمات المرور
- استخدام 2FA للحسابات الحساسة
- حماية Session Cookies
- إنهاء الجلسة عند تسجيل الخروج
- تسجيل محاولات الوصول غير المصرح بها
- اختبار كل وظيفة بحسابات مختلفة الصلاحيات

---

## خلاصة الدرس

هذا الدرس مهم جدًا لأنه يوضح أن تسجيل الدخول وحده لا يكفي. التطبيق يجب أن يعرف هوية المستخدم أولًا، ثم يتحقق من صلاحياته في كل طلب. كثير من الثغرات الخطيرة تحدث بسبب نسيان فحص الصلاحيات على الخادم.
